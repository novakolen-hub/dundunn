A horizontal strip below the GP banner that answers: "What can I trade right now?" Shows filtered setup types valid for the current sub-mode, with count of matching tickers.

│  Blocked: Momentum · BigDog · GapGo · OpeningDrive · ...     │
└──────────────────────────────────────────────────────────────┘

Data source

The backend getActiveSetups() returns this data, but calling it on every poll is expensive. Instead: include the validGP arrays in the poll response (via setupTypes in getPreTradeData()) and filter client-side. The dashboard already knows the current sub-mode from the poll. Client-side filtering = zero extra API cost.

Poll response
→

render bar

Implementation

Ship a static copy of SETUP_REGISTRY (just validGP, minRVOL, nature, label) to the dashboard as a JS constant. Run the same _checkSetupGP matching logic client-side. Green badge = allowed, yellow = caution, grayed = blocked. Clicking an allowed setup scrolls to the section containing matching tickers.

~60 lines new. JS constant + rActiveSetups() function + new div. No API calls.

C Setup Labels on Ticker Cards P1 MOD

What it is

Each ticker card in the In Play, Watching, Focused sections gets a small badge showing the matched setup type when one is detected. Caution (△) setups show a yellow warning. Blocked setups show a dimmed label with strikethrough.

Data source

The scanner already writes the setup field to each ticker's quote data (visible in _tq[ticker].setup). The _matchSetupType() logic runs server-side. The dashboard needs to:

  1. Read setup from the quote data (already available)
  2. Cross-reference against the client-side SETUP_REGISTRY copy
  3. Check current sub-mode to determine if it's allowed/caution/blocked
  4. Render the badge on the card

Badge examples

  [Momentum ✗] — dimmed/struck, not valid for C-Recovery

  [ReturnPullback ✗] — dimmed, bench mode blocks everything

D Pre-Trade Scorecard Modal P2 NEW

What it is

When you click a ticker's setup badge (or a dedicated "Evaluate" button), a modal overlay opens showing the full pre-trade analysis: 8-check scorecard with traffic lights, position sizing calculation, and a trade plan form that logs to the Trades tab.

Flow

Click ticker
→
fetch getPreTradeData(ticker)
→
render scorecard
→
user fills entry/stop/target
→
computePositionSize()
→
logPlannedTrade()

Modal sections

Section 1 — Ticker Context (auto-filled): Last price, VWAP distance, RVOL, key levels, confluence score, sector, beta, IVR. All from getPreTradeData() return.
Section 2 — Scorecard (8 checks): Traffic light grid. Each check shows GREEN/YELLOW/RED with the reason string. Pass count (e.g. "6/8 GREEN") determines overall signal. This calls evaluateScorecard() server-side with the selected setup type and side.
Section 3 — Setup Selection: Dropdown of setup types. Pre-selected based on the card's matched setup. Changing the dropdown re-runs the scorecard live (Check 1 changes based on setup type vs sub-mode). Shows validGP status, minRVOL check, session check.
Section 4 — Size Calculator: User inputs entry price, stop price, target price. Auto-calculates: position size (shares + dollars), dollar risk, R-multiples (1R/2R/3R), risk as % of account, scale plan from registry. All from computePositionSize(). Sub-mode sizing is automatic — C-Recovery shows $3-5K base, not A-level.
Section 5 — Log Trade: Button that calls logPlannedTrade() to write the plan to the Trades tab with a context snapshot. This is the bridge between "I'm evaluating" and "I'm executing."

API calls required

Two new fetch calls triggered by user interaction (not polling):

1. GET ?mode=pretrade&ticker=NVDA
   → calls getPreTradeData('NVDA')
   → returns: price, levels, context, sizing, setupTypes, sub-mode

2. POST ?mode=scorecard
   Body: { ticker, setupType, side, entry, stop, target }
   → calls evaluateScorecard() + computePositionSize()
   → returns: checks[], sizing{}, planned trade ID
Backend gap: The scanner's doGet() currently handles mode=dashboard and mode=pretrade and mode=scorecard. Need to verify that mode=pretrade and mode=scorecard are wired in v90.9's doGet(). If not, add two route handlers (~20 lines each).
~200 lines new. Modal HTML/CSS + 2 fetch calls + form logic. Biggest component.

E TM Book Integration P3 WIRE

What it is

Replace the current Book section's TradeStation-direct position data with Trade Management module data. The TM Trades tab has richer data: entry context, planned vs actual, scorecard at entry, P&L attribution, and daily limit tracking.

What changes

Position cards: Add fields from TM: setup type used, scorecard pass count at entry, planned entry vs actual, scale plan status (Tier 1 of 3, etc.), context snapshot link.
Daily limit bar: New horizontal bar at top of Book showing: daily P&L vs daily stop limit, trade count vs max trades, exposure vs max exposure. Data from getDailyLimitStatus(). Color-coded: green (under 50%), yellow (50-80%), red (>80%).
Sync status indicator: Small badge showing last syncTrades() timestamp and whether there are unmatched executions. Warns if sync is stale (>5 min).

Data source

Include TM data in the dashboard poll response. Add to getTimerState() or create a lightweight getTMSummary() that returns: open trades array (from Trades tab), daily limit status, last sync time, unmatched execution count.

Quota consideration: Reading from the Trades tab on every 60-second poll adds ~2-3 Sheets reads per cycle. At 420 cycles/day (7 hours), that's ~1,260 extra reads. Well within the 20K daily limit, but should be gated to market hours only.
~120 lines changed in rBook() + new daily limit bar + backend summary endpoint.
